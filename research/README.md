# Research Index

Investigative findings produced before implementation of the frontend design system.
These are not operational protocols — see `guidance/` for binding implementation rules.
They inform decisions; they do not prescribe them.

---

## How this was produced

All documents were generated by delegated agents during a pre-implementation investigation
session (2026-03-18). The orchestrator provided scoped prompts and source material;
agents conducted independent research and wrote findings directly. No synthesis was
done across documents — that is a separate step.

Agent model used per document is noted below. Haiku was used for mechanical extraction
(PDF parsing, file reading, cross-referencing); Sonnet for tasks requiring judgment
(quality analysis, pattern research).

## Documents

### `brand-guidelines.md`
**Source:** Official "Саме Те Guidelines" PDF (`same-te-landing/Саме Те Guidelines.pdf`)
**Agent:** Haiku — installed `poppler` to extract PDF text, read all pages
**Worth:** The only authoritative source for the formal brand specification. Covers
tone of voice, color palette with hex values, typography hierarchy, logo symbolism,
clear-space rules, and explicit prohibited uses. Primary source — quote it, don't
paraphrase it. Note: the PDF frames the audience as "teenagers" (підлітки); the mail
club's actual participants are vetted young adults. Visual system is binding; copy tone
is directional.

### `font-system.md`
**Source:** `same-te-landing/fonts/` directory — woff2 files and full TTF family
**Agent:** Haiku — read `index.html`, listed all font files, cross-referenced usage
**Worth:** Documents why only 3 of 18+ available font files were chosen for the
landing page (Regular + SemiBold from Mont, Dark from CyGrotesk). Explains the
OTF→woff2 conversion that happened for CyGrotesk. Includes a complete weight table,
CSS `@font-face` declarations ready to copy, and a note on what additional weights
the app might need. Actionable immediately — copy the 3 woff2 files and the CSS.

### `logo-system.md`
**Source:** `same-te-landing/Logo/SVG/` — all 24 SVG variants, plus 3 root SVGs
**Agent:** Haiku — read every SVG file, matched root files to their source variants
**Worth:** Maps the full logo variant system (4 categories × 6 colors = 24 files).
Confirms which files ended up in the landing page and why. Critical finding: the
**Secondary Horizontal** variants (mark + wordmark side-by-side) were never used in
the landing page but are the correct choice for a web app nav header. Answers the
"what happened to the SVGs" question — they were selectively copied from Logo/SVG/
to the project root, renamed.

### `tailwind-setup.md`
**Source:** cargo-leptos source code (GitHub), Tailwind v4 official docs, v4 release blog
**Agent:** Sonnet — web research, read cargo-leptos source directly to verify behavior
**Worth:** Authoritative setup guide for Tailwind v4 native binary in this project.
Confirmed from source: cargo-leptos downloads the binary automatically on first build
(no manual install needed), uses v4.2.1 by default, requires `tailwind-input-file` in
`Cargo.toml` to activate. Critical finding: `@source "../src"` is required — v4
auto-detection does not reliably reach `src/` from `style/tailwind.css`. Also documents
the dynamic class constraint and how Leptos `class:` syntax interacts with the scanner.
Minimal working setup is two additions to the codebase.

### `css-patterns.md`
**Source:** Tailwind v4 docs, Adam Wathan's utility-first essay, Josh Comeau, Lea Verou,
CSS-Tricks, MDN, Every Layout, web.dev forms accessibility, Open Props, moderncss.dev
**Agent:** Sonnet — web research with explicit instruction to skip SEO blogs and tutorials
**Worth:** The most directly actionable document for implementation. Covers: two-tier
token design (`@theme` for raw tokens, `:root` for semantic aliases), the Leptos
component as the abstraction layer (not `@apply`), CSS-equivalent of cva via
`data-*` attributes + CSS custom property hooks, admin density via layout context tokens,
`focus-visible` vs `focus`, OTP/phone input specifics, and a class ordering convention.
Contains full working CSS and Rust snippets for buttons (5 states × 3 variants), badges,
and form fields. Read this before writing any component CSS.

### `landing-page-analysis.md`
**Source:** `same-te-landing/index.html` — static source + live Chrome rendering
**Agent:** Sonnet — two passes: static analysis, then Chrome visual inspection
(Chrome pass required a workaround: `python3 -m http.server` because the navigate
tool prepends `https://` to `file://` URLs)
**Worth:** Two-part document. Static analysis: thorough code and accessibility audit —
3 WCAG AA contrast failures (CTA button, label, footer text), missing `<h1>`, steps
as `<div>` instead of `<ol>`, 5 hardcoded color values bypassing the token system, tag
rotation fragility, `z-index: 100` on a decorative layer. Also documents what was done
well and should be carried forward verbatim. App-scale concerns section is particularly
valuable — lists 7 patterns from the landing page that will cause problems in the app
if adopted uncritically (z-index, literal color names, no spacing scale, no form styles,
`overflow-x: hidden` on body, no global `:focus-visible`, steps as divs).
Chrome visual pass confirmed all three contrast failures with live computed values,
and added three new findings not catchable from static analysis:
- **Step heading clipping is real and measurable:** all four `.step h3` elements show
  `scrollHeight: 23px > clientHeight: 21px` — 2px of glyph overflow on every heading.
  `line-height: 1` at 20.8px is definitively too tight for CyGrotesk Cyrillic.
- **Grain is near-invisible on dark surfaces:** `mix-blend-mode: multiply` at 0.09
  opacity on `#161616` produces negligible effect. The grain reads as texture on light
  sections; on dark sections it barely registers.
- **CTA button is below the fold at 1280×800:** hero height is 623.7px; the button
  and scroll-hint are both off-screen on first load at common laptop viewport height.
